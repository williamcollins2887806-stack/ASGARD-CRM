## КОНЦЕПЦИЯ

Рабочий проходит несколько оплачиваемых этапов ДО выхода на объект: медосмотр, дорога, склад, ожидание. Каждый этап оплачивается по тарифной сетке (утв. Кудряшов О.С. 01.10.2025). Сейчас в Field есть только «Начать смену» на объекте — всё что до этого никак не учитывается.

### Ключевые принципы

1. **Этапы — это конструктор, не жёсткая цепочка.** РП при назначении выбирает какие этапы нужны конкретному человеку. Кто-то местный — ему не нужна дорога. Кто-то прошёл медосмотр месяц назад — этап «медосмотр» не нужен.

2. **Ожидание — ТОЛЬКО вне дома.** Ожидание оплачивается только когда человек находится не у себя дома (уже приехал куда-то: на склад, в транзитный город, на базу). Медосмотр дома + 2 недели дома до вылета = 0 дней ожидания. Приехал на склад, отработал день, ждёт отправки 3 дня = 3 дня ожидания.

3. **Медосмотр отвязан от командировки.** Может быть пройден за месяц до выезда. Это отдельный оплачиваемый этап, не привязанный к дороге/ожиданию.

4. **Мастер = глаза и руки РП.** Мастер видит ВСЁ то же что РП: все этапы всех рабочих своей бригады. Может запросить корректировку у РП. Не все рабочие владеют телефоном — мастер должен иметь возможность вести учёт за них.

5. **Защита от обмана, но без блокировки работы.** Система предупреждает об аномалиях, но не блокирует. РП принимает финальное решение.

## НОВАЯ ТАБЛИЦА

Миграция **V062__field_trip_stages.sql**:

```sql
CREATE TABLE IF NOT EXISTS field_trip_stages (
    id SERIAL PRIMARY KEY,
    employee_id INTEGER NOT NULL REFERENCES employees(id),
    work_id INTEGER NOT NULL REFERENCES works(id),
    assignment_id INTEGER REFERENCES employee_assignments(id),

    -- Тип этапа
    stage_type VARCHAR(30) NOT NULL,
    -- 'medical'       — медосмотр
    -- 'travel'        — дорога (любой транспорт)
    -- 'waiting'       — ожидание (ТОЛЬКО вне дома)
    -- 'warehouse'     — работа на складе
    -- 'day_off'       — выходной в командировке
    -- 'object'        — объект (связь с field_checkins)

    -- Дата (один этап = один день ИЛИ диапазон дней)
    date_from DATE NOT NULL,
    date_to DATE,                      -- NULL = один день
    days_count INTEGER DEFAULT 1,      -- кол-во дней (рассчитывается)

    -- Тариф
    tariff_id INTEGER REFERENCES field_tariff_grid(id), -- ссылка на конкретный тариф
    tariff_points INTEGER NOT NULL,
    rate_per_day DECIMAL(10,2) NOT NULL,
    amount_earned DECIMAL(12,2),       -- = days_count × rate_per_day (или days_approved × rate_per_day)

    -- Детали (JSONB, зависят от типа)
    details JSONB DEFAULT '{}',
    -- medical: { "clinic": "...", "conclusion_photo": "filename.jpg" }
    -- travel:  { "transport": "plane|train|bus|car", "route": "Москва → Кемерово",
    --            "flight": "S7-2541", "departure_time": "08:30", "arrival_time": "14:30" }
    -- waiting: { "location": "Кемерово", "reason": "Ожидание транспорта на объект" }
    -- warehouse: { "warehouse_name": "Склад Асгард, Москва" }
    -- day_off: { "reason": "Воскресенье" | "Карантинный день" | "Нерабочий день" }

    -- Привязка к логистике (билет → дорога)
    logistics_id INTEGER REFERENCES field_logistics(id),

    -- Источник отметки
    source VARCHAR(20) DEFAULT 'self',
    -- 'self'    — рабочий сам отметил
    -- 'master'  — мастер отметил за рабочего
    -- 'pm'      — РП проставил вручную
    -- 'auto'    — система (привязка к билету)
    source_employee_id INTEGER,        -- кто именно отметил (если master)

    -- Статус и подтверждение
    status VARCHAR(20) DEFAULT 'active',
    -- 'planned'   — РП запланировал, рабочий ещё не начал
    -- 'active'    — в процессе
    -- 'completed' — рабочий/мастер завершил
    -- 'approved'  — РП подтвердил
    -- 'adjusted'  — РП скорректировал (days_approved ≠ days_count)
    -- 'rejected'  — РП отклонил (не оплачивается)

    days_approved INTEGER,             -- если РП скорректировал кол-во дней
    approved_by INTEGER,
    approved_at TIMESTAMP,
    adjustment_note TEXT,              -- причина корректировки

    -- Фото-подтверждение (опционально)
    photo_filename VARCHAR(255),
    photo_original VARCHAR(255),

    -- Мета
    note TEXT,                         -- комментарий рабочего/мастера
    created_by INTEGER,                -- кто создал запись
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_trip_stages_emp_work ON field_trip_stages(employee_id, work_id);
CREATE INDEX IF NOT EXISTS idx_trip_stages_work_date ON field_trip_stages(work_id, date_from);
CREATE INDEX IF NOT EXISTS idx_trip_stages_status ON field_trip_stages(work_id, status);

-- Уникальность: один сотрудник не может иметь два активных этапа одного типа на один день
CREATE UNIQUE INDEX IF NOT EXISTS idx_trip_stages_unique_day
    ON field_trip_stages(employee_id, work_id, stage_type, date_from)
    WHERE status NOT IN ('rejected');
```

**НЕ создавать отдельных таблиц для каждого типа этапа** — одна таблица, тип в `stage_type`, детали в `details` (JSONB).

## API ENDPOINTS

### Новый файл: `src/routes/field-stages.js`

**Prefix:** `/api/field/stages`

#### Для РП (CRM auth — fastify.authenticate):

```
POST /
  Body: { employee_id, work_id, stage_type, date_from, date_to?,
          tariff_id?, details?, note? }
  → Создать этап вручную (РП проставляет за рабочего)
  → source='pm', created_by=user.id
  → Автоматически рассчитать days_count, rate_per_day (из tariff_grid), amount_earned
  → Если tariff_id не указан — подобрать автоматически из field_tariff_grid по stage_type:
    medical → category='special', position LIKE '%осмотр%'
    travel → category='special', position LIKE '%Дорога%'
    waiting → category='special', position LIKE '%Выходной%' (те же 6 баллов)
    warehouse → category='warehouse' (тариф склада из assignment)
    day_off → category='special', position LIKE '%Выходной%'

GET /project/:work_id
  → Все этапы всех сотрудников проекта
  → JOIN employees (fio), field_tariff_grid (position_name)
  → Группировка по сотрудникам
  → Итоги: total_days, total_earned по каждому сотруднику

GET /project/:work_id/calendar
  → Календарная сетка: сотрудник × дата → тип этапа + статус
  → Для отображения цветной таблицы в desktop CRM
  → Query params: date_from, date_to (период)

GET /employee/:employee_id/work/:work_id
  → Все этапы конкретного сотрудника на проекте
  → Хронологический порядок
  → Включает amount_earned, status, source

PUT /:id/approve
  Body: { days_approved?, adjustment_note? }
  → РП подтверждает этап
  → Если days_approved указан и ≠ days_count → status='adjusted', пересчёт amount_earned
  → Если не указан → status='approved', days_approved=days_count

PUT /:id/reject
  Body: { adjustment_note }
  → status='rejected', amount_earned=0

PUT /:id
  Body: { date_from?, date_to?, details?, note?, stage_type? }
  → РП редактирует этап (только planned/active/completed, НЕ approved)

DELETE /:id
  → Удалить этап (только planned, НЕ active/completed)

POST /bulk
  Body: { employee_ids: [1,2,3], work_id, stage_type, date_from, date_to?, details? }
  → Массовое создание этапов (например: всей бригаде проставить выходной)
```

#### Для мастера (Field auth — fastify.fieldAuthenticate, field_role IN ('shift_master','senior_master')):

```
GET /my-crew/:work_id
  → Все этапы всех рабочих бригады мастера (аналог GET /project/:work_id, но фильтр по смене мастера)
  → Мастер видит ВСЁ то же что РП по своим людям

POST /on-behalf
  Body: { employee_id, work_id, stage_type, date_from, date_to?, details?, note? }
  → Мастер создаёт этап ЗА рабочего (source='master', source_employee_id=мастер)
  → Валидация: employee_id должен быть в бригаде мастера

PUT /on-behalf/:id
  Body: { date_to?, details?, note? }
  → Мастер обновляет этап рабочего (завершает, добавляет фото)

POST /request-correction
  Body: { stage_id, requested_days?, note }
  → Мастер запрашивает корректировку у РП
  → Создаёт уведомление РП: "Мастер {имя} просит скорректировать этап {тип} для {рабочий}: {note}"
  → НЕ меняет данные — только уведомление
```

#### Для рабочего (Field auth — fastify.fieldAuthenticate):

```
GET /my/:work_id
  → Мои этапы на проекте (хронологически)
  → Включает amount_earned, status

POST /my/start
  Body: { work_id, stage_type, details? }
  → Начать этап (source='self')
  → date_from = TODAY
  → Валидации:
    - Нельзя начать два этапа одного типа в один день
    - Нельзя начать 'object' (для объекта используется /checkin)
    - 'waiting' — нельзя начать самостоятельно если РП не включил этот этап

POST /my/end
  Body: { stage_id, photo_filename?, note? }
  → Завершить этап
  → date_to = TODAY, рассчитать days_count и amount_earned
  → Для medical — фото заключения обязательно (photo_filename)

GET /my/current/:work_id
  → Текущий активный этап (status='active') или null
  → Используется на главной для отображения: "Сейчас: Дорога ✈️ · день 1"
```

- Кнопка: Массовое действие (выбрать несколько сотрудников → добавить всем выходной/ожидание)

### Предупреждения (аномалии)

Подсвечивать жёлтым:
- Медосмотр > 3 дней
- Ожидание > 5 дней
- Дорога > 3 дней (для внутренних рейсов)
- Этап без подтверждения > 48ч

## ИНТЕГРАЦИЯ С СУЩЕСТВУЮЩИМИ МОДУЛЯМИ

### Связь с field_logistics (билеты)

Когда офис-менеджер добавляет билет через field_logistics:
- Если тип `ticket_to` или `ticket_back` — автоматически создать этап `travel` (status='planned') с details из билета
- date_from = дата вылета, date_to = дата прилёта
- logistics_id = id билета
- Рабочий видит запланированную дорогу и может нажать «Выехал»

### Связь с field_checkins (смены)

Этап `object` не создаётся вручную — он виртуальный. Когда рабочий нажимает «Начать смену» через существующий /checkin — это автоматически означает что все предыдущие этапы завершены. Если есть незавершённые этапы (active) — они автоматически завершаются (date_to = вчера).

### Связь с money.js (финансы)

## КОДОВЫЕ КОНВЕНЦИИ (напоминание)

### Backend
- `async function routes(fastify, options) { ... } module.exports = routes;`
- Для РП: `preHandler: [fastify.authenticate]` + проверка ролей
- Для мастера/рабочего: `preHandler: [fastify.fieldAuthenticate]`

- SQL: параметризованные `$1, $2` — никакого string concat
- Ошибки: `{ error: 'Текст по-русски' }`

- Данные через `setTimeout(() => load(), 0)` — fire-and-forget

- Цвета ТОЛЬКО через `DS.t.*`
- Шрифты через `DS.font('scale')`

## ВАЖНО — НЕ ЗАБЫТЬ

- [ ] Мастер видит ВСЁ то же что РП по своим людям (полный доступ к этапам бригады)

- [ ] Медосмотр НЕ привязан к дороге — может быть за месяц до выезда

- [ ] При начале смены (checkin) — незавершённые этапы автоматически закрываются
- [ ] Автосоздание этапа travel при добавлении билета
